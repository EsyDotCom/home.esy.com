# Workflow Manifest Standard

> Created: May 28, 2026
> Status: **Active** — Phase 1 (manifest as source of truth)
> Scope: Cross-repo (`esy.com`, `app.esy.com`, `api.esy.com`)

## Purpose

Every Esy workflow template is **versioned, model-pinned, and immutable**. A
published artifact must carry a complete, reproducible record of the workflow
version and the exact models that produced it. This standard defines the
**manifest** — the single source of truth for what a workflow version does —
and the discipline that keeps the marketing claim ("each version pins its
models") structurally true rather than decorative copy.

The motivating problem: before this standard, the clip-art artifact spec card
showed `Engine: gpt-image-2` as a hand-typed string in
`src/data/clip-art-artifacts.ts`. Nothing connected that string to what
actually ran, and nothing prevented it from drifting. The manifest closes that
gap.

## The pin

A real pin has two parts (per the *Encyclopedia of Agentic Coding Patterns*,
"Pinning"):

1. **A human-readable label** — `v1.0.0`, `gpt-image-2`.
2. **An immutable identifier** — a SHA-256 content hash of the manifest, and a
   dated/qualified model snapshot (`gpt-image-2-2026-04-21`), never a floating
   alias (`latest`, `chatgpt-image-latest`).

Aliases like `latest` or `canonical` are **not pins** — they are movable
pointers that resolve to whatever they point at today. A pin is durable enough
that drift has to announce itself.

## What a manifest pins

A manifest is the smallest unit of behavior you can reproduce, evaluate, or
roll back. It pins **the entire decision tree**, not just a headline model:

- **Input contract** — the parameters a run accepts and their allowed values.
- **Pipeline** — the ordered stages.
- **Per-stage policy** — for each stage, either a single pinned model/template
  or a **routing table** (input-conditioned model selection). The routing rules
  themselves are pinned, so the same input always resolves to the same model.
- **Output contract** — artifact type, formats, storage convention.

> A workflow version pins **how every model decision gets made**, not just
> which models exist. Given the same input, the same version always produces
> the same routing and the same provider calls.

## File layout

```
src/data/workflow-manifests/
  {workflow-slug}.{version}.json        ← the manifest (source of truth)
  {workflow-slug}.{version}.lock.json   ← { version, sha256 } committed lockfile
```

The manifest is committed JSON. The lockfile holds the SHA-256 computed over
the canonical (key-sorted) serialization of the manifest. The lockfile is
regenerated by `scripts/hash-workflow-manifests.mjs` and committed alongside
the manifest. Editing a manifest without regenerating its lock is caught by the
hash check (`npm run manifests:check`).

### Why a committed lockfile instead of runtime hashing

`esy.com` is a static export (`output: export`) and the clip-art detail page is
a client component. Importing `node:crypto` into that path breaks client
bundling. So the hash is computed at author/CI time by a Node script and
committed as data the TS loader reads — exactly like `package-lock.json`.

## Manifest schema

```jsonc
{
  "workflow": "generate-clip-art-asset",   // slug, stable across versions
  "version": "v1.0.0",                      // semver-flavored, immutable
  "title": "Generate Clip Art Asset",       // display name
  "released_at": "2026-05-15T00:00:00Z",
  "status_note": "...",                     // optional human note

  "input_contract": {
    "prompt":  { "type": "string", "required": true },
    "style":   { "type": "enum", "values": ["cartoon","chibi","illustrated","watercolor","flat"] },
    "aspect":  { "type": "enum", "values": ["1:1","3:4","4:3","16:9"] }
  },

  "pipeline": [
    {
      "stage": "resolve_prompt",
      "kind": "prompt-template",
      "prompt_template": "clip-art-prompt@v1"
    },
    {
      "stage": "render_image",
      "kind": "model",
      "model": { "provider": "openai", "id": "gpt-image-2", "snapshot": "gpt-image-2-2026-04-21" }
    },
    {
      "stage": "remove_background",
      "kind": "model-routed",
      "routing": [
        { "when": { "style": ["cartoon","chibi","illustrated","watercolor","flat"] },
          "model": { "provider": "fal", "id": "birefnet-light", "snapshot": null } }
      ]
    },
    {
      "stage": "store",
      "kind": "storage",
      "destination": "r2://artifacts/clip-art/{artifact_id}/"
    }
  ],

  "output_contract": {
    "artifact_type": "clip-art",
    "formats": ["WEBP", "PNG"],
    "storage_key": "artifacts/{artifact_type}/{artifact_id}/processed.webp"
  }
}
```

### Stage kinds

| `kind` | Meaning | Resolves to |
|--------|---------|-------------|
| `prompt-template` | Deterministic prompt assembly from a pinned template id | a `prompt_template` id |
| `model` | One pinned model for all inputs | a `model` object |
| `model-routed` | Input-conditioned model selection | a `model` object, chosen by the first matching `routing[].when` |
| `storage` | Durable write | a storage destination |

### Model object

```jsonc
{ "provider": "openai", "id": "gpt-image-2", "snapshot": "gpt-image-2-2026-04-21" }
```

- `provider` + `id` + `snapshot` together are the pin.
- `snapshot` MAY be `null` in the **marketing manifest** when the provider
  snapshot is pinned by the execution plane (`api.esy.com`) and recorded in the
  run record. The marketing manifest declares intent; the run record records
  the exact resolved snapshot. (See "Division of ownership.")

## Versioning + aliases

Versions are **immutable**. You never mutate `v1.0.0`; you publish `v1.1.0`.
Promotion is **alias movement**, not version mutation (per MLflow / model
registry best practice: *immutable versions, mutable aliases*).

| Bump | When | Example |
|------|------|---------|
| **Patch** (`v1.0.0 → v1.0.1`) | Same model family, newer dated snapshot, no behavioral drift | `gpt-image-2-2026-04-21` → `gpt-image-2-2026-06-15` |
| **Minor** (`v1.0.0 → v1.1.0`) | Model swap in the same quality/cost class with a documented behavior diff | swap BG remover within the same tier |
| **Major** (`v1.0.0 → v2.0.0`) | Material change to style, cost, or output shape | `gpt-image-2` → a different image model |

Aliases are movable labels that point at a version:

| Alias | Who sees it | Cardinality |
|-------|-------------|-------------|
| `canonical` | All users (the default) | exactly 1 |
| `available` | Users who opt into a permanent alternative | 0–3 |
| `shadow` | Operators only — runs in parallel, output never shown | 0–5, each with an expiry |
| `deprecated` | Anyone reproducing an old run; no new opt-ins | unbounded, no UI surface |

### Bump a version vs spawn a sibling workflow

- **Bump the version** when it's the same job — same input/output contract,
  same intent — served by a different model decision.
- **Spawn a sibling workflow** when the *output identity* changes. A
  research-assisted variant is `Generate Historically Referenced Clip Art`, not
  `v2` of this workflow.

> Rule: if a user would meaningfully prefer one over the other based on the
> output, they're siblings. If one is strictly better at the same job, it's a
> version bump.

## Experiments without explosion

The combinatorial space of model combinations is unbounded. The **catalog is
hand-curated**. Most combinations you *could* build never become versions —
they live as internal runs you compare and discard.

Per-workflow budget:

- **1** canonical (always)
- **0–3** available alternatives (each earns its slot by serving a distinct
  artifact identity)
- **0–5** shadow/experimental at a time (each has an expiry — promote, demote,
  or kill within a fixed window)
- unbounded deprecated (no UI surface; exist only for run reproducibility)

Gate question before publishing any version: **what user-facing artifact
identity does this serve?** If the answer is "this combination exists," it's an
internal run, not a version.

## Run records

Each run records the resolved pin so the artifact is reproducible:

```jsonc
{
  "workflow": "generate-clip-art-asset",
  "version": "v1.0.0",
  "manifest_sha256": "…",            // from the lockfile at run time
  "alias_at_run": "canonical",
  "inputs": { "style": "chibi", "aspect": "1:1" },
  "resolved": {
    "render_model": "gpt-image-2-2026-04-21",
    "background_removal_model": "birefnet-light",
    "prompt_template": "clip-art-prompt@v1"
  },
  "resolved_prompt": "Chibi-style clip art of …",
  "provider_response": { /* headers, request ids, cost */ }
}
```

The orchestration runner already records `workflow.version`, hashes artifacts
and prompts (`hasher.js`), and writes durable `RUN.json` records — this
standard upgrades `version` from the placeholder `"local-dev"` to a real
manifest version + hash.

## Division of ownership

| Repo | Owns |
|------|------|
| `esy.com` (this repo) | The **published manifest** (representative source of truth for the marketing surface) + the drift guard that asserts published artifacts only cite models their manifest version declares. |
| `app.esy.com` | Workflow/version admin, alias promotion UI, version diff. |
| `api.esy.com` | The **resolver** (`resolve(slug, alias) → ResolvedManifest`), the **runtime enforcer** (calls providers only with manifest-resolved params, no `latest`, no env overrides), provider snapshot pinning, and the authoritative run record. |

The marketing manifest declares intent; the execution plane records exactly
what ran. The drift guard in this repo keeps the two from diverging silently.

## What Phase 1 implements (this repo)

1. Manifest format + a real pinned manifest for `generate-clip-art-asset@v1.0.0`.
2. Committed lockfile with the manifest SHA-256.
3. A TS loader/resolver (`src/lib/workflow-manifests/`) that is the single
   source of truth for the clip-art spec card's model/format fields.
4. Clip-art artifacts reference `workflowVersion` + record per-run `resolved`
   models instead of free-typed `engine`/`backgroundRemoval`/`formats`.
5. A build-time **drift guard** (`scripts/validate-artifact-provenance.mjs`)
   that fails the build if any artifact's resolved models are not declared in
   its referenced manifest version, or if a lockfile is stale.
6. Real provenance (workflow version + manifest hash) surfaced on the
   `/clip-art/[slug]` detail page.

### Explicitly deferred (other repos / later phases)

- The runtime resolver + enforcer in `api.esy.com` (Phase 2).
- Alias promotion as an audited operation + CLI (Phase 3).
- Shadow mode with comparison/evaluation artifacts (Phase 4).
- Version-history UI + manifest diff viewer in `app.esy.com` (Phase 5).
- Sub-workflows (shared, independently-versioned stages) — adopt only when a
  second template genuinely shares a stage.

## Industry alignment

This standard matches the 2026 consensus on agentic-workflow versioning:

- **Pin the whole stack as one hashed bundle** — model id, prompt, tools,
  policy — and start runs only with a pinned `version_id`, never `latest`
  (*Agent Patterns*, "Agent Versioning"; *Encyclopedia of Agentic Coding
  Patterns*, "Pinning").
- **Immutable versions, mutable aliases**; promotion is a policy-driven state
  machine with an audit log (MLflow Model Registry; Resilio model-registry
  best practices).
- **shadow → canary → canonical** progression for promotion; shadow output is
  logged, never shown (industry LLM rollout guides, 2026).
- **Simplicity discipline**: "add complexity only when it demonstrably improves
  outcomes" (Anthropic, *Building Effective Agents*).

What is distinctly Esy: workflow versions are **part of the public artifact
taxonomy**. A published artifact carries its workflow + version + pinned models
as visible provenance on the marketing site, not just as an internal ops
record. (Princeton's HAL benchmark — the same model scoring 64.9% vs 57.6% on
GAIA across two scaffolds — is the external evidence for Esy's engineer-first
claim that *how you compose the workflow matters more than which model you swap
in*.)
